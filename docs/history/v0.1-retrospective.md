# Sluice v0.1.0 Retrospective

Frozen record of what we planned, what we shipped, and what surprised us on the road to v0.1.0.

Written after release commit `d4c65c2` ("Release Sluice v0.1.0").

Historical document. Not maintained. See `docs/architecture/v0.2.md` for what's next.

---

## 1. What we planned

The v0.1 plan (as of `POC_PLAN.md` at draft time) was to ship, within roughly two weeks:

- Multi-upstream config with per-tool policy rules
- Tool-poisoning detector on `tools/list` responses
- SQLite audit sink + `sluice logs` CLI
- Session-scoped taint (in-memory, substring-based)
- Streamable HTTP transport per MCP 2025-03-26
- `${VAR}` expansion in config
- Latency benchmark harness + published p50/p95 numbers
- Apache 2.0 license, SECURITY.md, GitHub Actions CI
- PyPI publication
- 30-second demo GIF in the README
- One external user before calling it "done"

Exit criteria: fresh install works in under 5 minutes, blocks a believable attack, one external user who isn't us.

## 3. What surprised us

Things we didn't predict at planning time.

### 3.1 PyPI name friction

Package name `sluice` was taken; we published as `sluice-taint`. Repo, CLI binary, and PyPI project name now diverge. Documented for future reference — but this is the kind of thing that would have been useful to check on day one of choosing the name. Filed as ADR 0005.

### 3.2 Latency was better than budgeted

Original architecture budget was <1 ms p50 on the clean path. Actual is 0.02 ms — a full order of magnitude under. Pydantic validation and the pure-function detector protocol turned out to be much cheaper than expected. The published number is now itself a positioning asset.

### 3.3 Session ID for stdio wasn't obvious

Streamable HTTP gives you a session ID in the `Mcp-Session-Id` header. stdio doesn't. We deferred the question in the v0.1 architecture doc (§11); the shipped code synthesizes one per-process. That's fine for the intended stdio use case (one Claude Desktop or Cursor subprocess per user) but shouldn't survive contact with any multi-tenant model. Filed as ADR 0004.

### 3.4 Config schema changes required more validation than expected

Multi-upstream + `include:` (deferred to v0.2) + per-tool rules made the Pydantic models nontrivial. The migration path from v0's single-`upstream:` block was solved by writing a `config.yaml.upgraded` file the user can review — a design we didn't originally consider and should keep for future migrations.

### 3.5 Tool-poisoning detector was cheap to write

We expected this to be the hardest new detector. In practice, the Invariant research gave us a clear pattern set, and the detector was ~one afternoon. The taint work took much longer than budgeted, in comparison.

## 4. What we cut or deferred

Explicit list of scope decisions, with rationale.

| Cut / Deferred | Where it went | Rationale |
|---|---|---|
| Full SSE session lifecycle (reconnect, resume) | v0.2 §4.5 | Happy path works; reconnect matters only under real network flakiness, and we don't have real users yet |
| Field-level taint provenance | v0.3 | Substring taint delivers 90% of the demo value at 10% of the complexity |
| Web dashboard | v0.2 §4.2 | Not needed for a proxy demo; adds real UI surface |
| Docker image | v0.2 §4.3 | Doesn't unblock v0.1 users (pip install works) |
| Policy presets | v0.2 §4.1 | Users can copy from `examples/`; presets is a v0.2 UX polish |
| LLM-based detection | v0.3+ | Latency and cost incompatible with a JSON-RPC middleware |
| OpenTelemetry exporter | v0.3 | Local `sluice logs` is enough for individual users; OTel is enterprise-shape |
| Field-level policy DSL | Unspecified | YAML+glob covers current needs; revisit only if users hit a wall |
| Plugin API docs | v0.2 §4.6 | Entry-point mechanism exists in code; docs unblock external detectors |

## 5. Metrics we didn't hit and don't know yet

- **Install-to-first-block time on a fresh machine.** Not measured empirically. Should be <5 min based on the code path but hasn't been walked through by an external tester.
- **False-positive rate at population scale.** Zero data — we have no external users.
- **p95 overhead for the redact path.** Measured on our machine; not published in README yet. Should be, in the next README update.

## 6. Lessons for v0.2

Things we'd tell ourselves at v0.2 kickoff.

1. **Set the adoption timebox before starting the build.** We ran v0.1 as a build sprint and treated adoption as an afterthought. v0.2 should have an adoption sprint scheduled explicitly at the start.
2. **Latency budgets are worth publishing even before optimization.** The 0.02 ms number became a positioning asset. Same discipline for dashboard render times.
3. **Config migration is a first-class feature.** Every future schema change writes a `config.yaml.upgraded` and tells the user, same as v0 → v1 did.
4. **Detectors are cheap; taint is expensive.** Budget accordingly. If we add a fifth detector in v0.2 it should be a weekend; if we touch taint algo, budget a week.
5. **ADRs get written when the decision is made, not when the code lands.** We skipped ADRs for v0.1 and now have to backfill six of them from memory. Don't do this again.
