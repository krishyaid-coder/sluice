# Sluice v0.1 — Target Architecture

This document specifies the v0.1.0 architecture for Sluice. It is the contract between everyone writing code in this repo.

If a code change conflicts with this doc, the doc wins — or the doc gets updated in the same PR. No silent drift.

**Audience:** maintainers and serious contributors. Reads cold.

**Status:** v0.1 implemented. Open questions in §11 marked resolved where decided.

---

## 1. Goals and non-goals

### Goals for v0.1
- Drop-in proxy for Claude Desktop / Cursor / any MCP client, via stdio or HTTP
- Support the current MCP transport spec (stdio + streamable HTTP)
- Inspect every JSON-RPC message in both directions
- Block, redact, or flag based on declarative policy
- Track simple cross-tool data flow (taint v1)
- Persist an audit trail queryable from the CLI
- Be installable via `pip install sluice` and `docker run`
- Be extensible by external contributors without forking (detector plugin interface)

### Non-goals for v0.1
- LLM-based detection (deferred to v0.3+)
- Distributed / multi-process audit store
- Field-level taint provenance (deferred to v0.3)
- A web dashboard SPA (server-rendered HTML only, and only in v0.2)
- A policy expression language (YAML rules only)
- Telemetry of any kind

## 2. Module layout

Target package tree for v0.1. Items marked ~~strikethrough~~ are planned but not yet implemented.

```
sluice/
  __init__.py
  __main__.py                  # routes to cli.main
  cli/
    main.py                    # Typer app: init, serve, stdio, logs, doctor, version
  config/
    schema.py                  # Pydantic models for config.yaml
    loader.py                  # load + validate + legacy migration
  proxy/
    transports/
      base.py                  # Transport protocol
      http.py                  # HTTP + basic SSE stub
      stdio.py                 # stdio bridge
    router.py                  # multi-upstream dispatch
    pipeline.py                # inspect_request / inspect_response
    forwarder.py               # HTTP upstream client
    models.py
  detectors/
    base.py                    # Detector protocol + registry
    secrets.py
    pii.py
    tool_poisoning.py
  policy/
    engine.py                  # per-tool / per-upstream rule matching
  session/
    taint.py                   # session-scoped taint API
    store.py                   # in-memory session registry
  audit/
    sink.py                    # AuditSink ABC
    sqlite.py                  # default SQLite sink
    stdout.py                  # structlog passthrough + chained sink
  ~~bench/~~                   # planned: latency benchmarks for CI
```

**Rationale for the new boundaries:**

- `cli/` — separating CLI from `__main__` makes the command surface testable and lets us add subcommands without growing one file.
- `config/schema.py` — Pydantic models give us validation, defaults, and IDE autocomplete for free. Schema changes become diffs in one file.
- `proxy/transports/` — three transports now (stdio, HTTP, streamable HTTP), and possibly SSE later. A shared base class + per-transport file keeps `pipeline.py` clean.
- `proxy/router.py` — multi-upstream is the single biggest scope expansion in v0.1; it deserves its own module.
- `detectors/base.py` — defining a `Detector` protocol unlocks third-party detectors as a stable extension point.
- `audit/` — SQLite as the default sink, but behind an interface so a Postgres or OTel sink can land later without breaking callers.
- `bench/` — benchmarks are first-class. They run in CI and gate releases on regression.

## 3. Config schema (v0.1)

Designed for: multi-upstream routing, per-tool policy, future-proof field names.

```yaml
# config.yaml
version: 1                          # schema version, required

# Default proxy listen settings (only used when running `serve`)
proxy:
  host: "127.0.0.1"
  port: 4444

# List of upstreams. At least one required.
upstreams:
  - name: "filesystem"              # unique, used in policy + logs
    transport: "stdio"              # stdio | http | streamable_http
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    env: {}                         # optional env overrides

  - name: "github"
    transport: "streamable_http"
    url: "https://api.github.com/mcp"
    headers:
      Authorization: "${GITHUB_TOKEN}"   # ${VAR} substitution at load time

# Routing: how incoming requests pick an upstream.
# For stdio mode, default is the first upstream.
# For HTTP mode, the client picks via the URL path: /u/<name>
routing:
  default: "filesystem"

# Detectors: which are enabled and any per-detector config.
detectors:
  secrets:
    enabled: true
  pii:
    enabled: true
    redaction: true
  tool_poisoning:
    enabled: true
    severity_threshold: "medium"

# Policy: list of rules, evaluated top-to-bottom. First match wins.
policy:
  rules:
    - detector: "secrets.aws_access_key"
      action: "block"
    - detector: "secrets.*"
      upstream: "github"            # optional: scope to upstream
      tool: "*"                     # optional: scope to tool name
      action: "block"
    - detector: "pii.*"
      action: "redact"
    - detector: "tool_poisoning.*"
      action: "flag"
  default_action: "flag"            # for any detector hit not covered above

# Session taint (v1)
taint:
  enabled: true
  min_length: 12
  scope: "session"                  # session | process (v0.1 supports both)

# Audit
audit:
  sink: "sqlite"                    # sqlite | stdout | both
  sqlite:
    path: "~/.sluice/audit.db"
    retention_days: 30

# Logging (operational, separate from audit)
logging:
  level: "info"
  format: "json"                    # json | console
```

**Schema rules:**

- `version: 1` is required. Future schema changes bump this and trigger `config/migrate.py`.
- Unknown top-level keys → loader error (catches typos).
- Unknown nested keys → warning (forward-compat for plugins).
- `${VAR}` expansion happens at load time; unresolved vars error unless the field is optional.
- All paths support `~` expansion.

**Migration from v0 (current `config.yaml`):**

- Single `upstream:` block → wrapped into `upstreams: [{name: "default", ...}]`
- Implicit version → version: 1
- `loader.py` detects schema-less configs and runs `migrate.py` automatically, writing the upgraded file to `config.yaml.upgraded` for the user to review.

## 4. CLI surface

Single binary, subcommands. Modeled on tools that feel professional (`gh`, `kubectl`, `docker`).

```
sluice init [--force] [--path PATH]
sluice serve [--config PATH] [--host HOST] [--port PORT]
sluice stdio [--config PATH] [--upstream NAME]
sluice logs [--since DURATION] [--upstream NAME] [--action ACTION] [--tail] [--json]
sluice doctor [--config PATH]
sluice version [--json]
```

**Per-command contract:**

| Command | Purpose | Exit codes |
|---|---|---|
| `init` | Write a starter `config.yaml` in the current directory. Refuses to overwrite without `--force`. | 0 success, 1 file exists, 2 write error |
| `serve` | Run the HTTP / streamable-HTTP proxy. Foreground process. | 0 clean shutdown, 1 config error, 2 bind error |
| `stdio` | Run as a stdio bridge for one upstream. Used by Claude Desktop / Cursor. | 0 clean exit, 1 config error, 2 upstream spawn error |
| `logs` | Query the audit store. Defaults to last hour, all upstreams, human format. `--json` for piping. `--tail` follows. | 0 success, 1 no audit sink, 2 query error |
| `doctor` | Validate config, ping each upstream, report taint store status, dump version info. | 0 healthy, 1 warnings, 2 errors |
| `version` | Print version. `--json` includes Python version, MCP spec version, OS. | 0 |

**Global flags:**
- `--config PATH` (defaults to `./config.yaml` then `~/.sluice/config.yaml`)
- `--log-level {debug,info,warn,error}`
- `--no-color` / respects `NO_COLOR` env var

**Output conventions:**
- Human output is the default. `--json` is available on every read-only command.
- Errors go to stderr. Data goes to stdout. Always.
- Never print secrets in any output, even from the audit store — they're stored hashed/redacted.

## 5. Transport architecture

All transports implement one interface:

```python
class Transport(Protocol):
    async def run(self, pipeline: Pipeline, router: Router) -> None: ...
    async def shutdown(self) -> None: ...
```

The transport owns I/O. The pipeline owns inspection. The router owns upstream selection. **No transport contains policy logic.** This is the single most important architectural invariant — it's why we can add streamable HTTP without touching detectors.

### stdio
- One process, one upstream (selected via `--upstream`)
- Reads NDJSON from stdin, writes NDJSON to stdout
- Spawns the upstream as a subprocess (unchanged from current code)

### HTTP
- Single port, multiple upstreams
- Route is `POST /u/<upstream-name>` → dispatched by router
- Backward compat: `POST /` → routes to `routing.default`

### Streamable HTTP (MCP 2025-03-26)
- Supports both `POST /u/<name>` (single request/response) and `GET /u/<name>` (SSE stream for server-initiated messages)
- Session ID propagated via `Mcp-Session-Id` header
- Pipeline gets the session ID for taint scoping (see §7)

Adding a new transport later (e.g. WebSocket) requires:
1. A new file under `proxy/transports/`
2. Implementing `Transport`
3. A registry entry
4. No changes elsewhere

## 6. Multi-upstream routing

`Router` is the dispatcher between transport and upstream.

```python
class Router:
    def __init__(self, upstreams: dict[str, UpstreamConfig]): ...
    async def dispatch(
        self,
        upstream_name: str,
        message: JsonRpcMessage,
        session_id: str,
    ) -> JsonRpcMessage: ...
```

Responsibilities:
- Resolve `upstream_name` to a connection (lazy-spawn stdio, pooled HTTP client for streamable)
- Forward the request, await response
- Surface connection errors as JSON-RPC errors back to the agent (never crash the proxy)
- Track per-upstream health for `doctor` and the audit log

The pipeline calls the router; the router does not call detectors. Order: `transport → pipeline.inspect_request → router.dispatch → pipeline.inspect_response → transport`.

## 7. Audit store

### Goals
- Queryable via CLI (`sluice logs`)
- Survives process restart
- Bounded growth (retention policy)
- Never stores raw secrets

### SQLite schema

```sql
CREATE TABLE events (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    ts          INTEGER NOT NULL,                  -- unix ms
    session_id  TEXT NOT NULL,
    upstream    TEXT NOT NULL,
    direction   TEXT NOT NULL,                     -- 'request' | 'response'
    method      TEXT,                              -- JSON-RPC method
    tool        TEXT,                              -- tools/call name, if applicable
    action      TEXT NOT NULL,                     -- 'pass' | 'block' | 'redact' | 'flag'
    detectors   TEXT,                              -- JSON array of detector ids that fired
    rule        TEXT,                              -- policy rule id, if any
    redacted_preview TEXT,                         -- safe preview, no raw secrets
    latency_us  INTEGER
);

CREATE INDEX idx_events_ts ON events(ts);
CREATE INDEX idx_events_upstream_ts ON events(upstream, ts);
CREATE INDEX idx_events_session ON events(session_id);

CREATE TABLE taint_marks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id  TEXT NOT NULL,
    hash        TEXT NOT NULL,                     -- sha256 of tainted value
    first_seen  INTEGER NOT NULL,
    source_event INTEGER REFERENCES events(id)
);

CREATE INDEX idx_taint_session_hash ON taint_marks(session_id, hash);
```

**Key invariants:**
- `redacted_preview` is `aws_access_key=AKIA****EXAMPLE` style — visually informative, cryptographically useless.
- Raw secret values never enter SQLite. Taint matches are stored as sha256 hashes.
- Retention runs lazily on writes: opportunistic delete of rows older than `retention_days`.

### Sink interface

```python
class AuditSink(Protocol):
    async def write(self, event: AuditEvent) -> None: ...
    async def query(self, filter: AuditFilter) -> AsyncIterator[AuditEvent]: ...
    async def close(self) -> None: ...
```

Default sink: `SqliteSink`. Dev sink: `StdoutSink` (structlog). `audit.sink: both` chains them.

## 8. Detector plugin interface

This is the OSS extension point. External contributors should be able to ship a detector as a separate package without touching Sluice core.

### Protocol

```python
from typing import Protocol

class Detector(Protocol):
    id: str                       # e.g. "secrets.aws_access_key"
    severity: Literal["low", "medium", "high", "critical"]

    def scan(self, text: str, context: ScanContext) -> list[Hit]: ...

@dataclass(frozen=True)
class ScanContext:
    direction: Literal["request", "response"]
    method: str | None
    tool: str | None
    upstream: str

@dataclass(frozen=True)
class Hit:
    detector_id: str
    start: int
    end: int
    matched: str                  # raw match, used only inside the proxy
    label: str                    # human-readable, safe to log
```

### Registration

Built-in detectors register via a `@register` decorator in `detectors/base.py`.

Third-party detectors register via Python entry points:

```toml
# in a contributor's pyproject.toml
[project.entry-points."sluice.detectors"]
my_org_secrets = "my_org_sluice_extras:MyOrgSecretsDetector"
```

`loader.py` discovers entry points at startup and merges them into the registry.

### Stability contract
- `Detector`, `ScanContext`, `Hit` are stable across v0.x.
- Adding optional fields to `ScanContext` is non-breaking.
- Removing or changing semantics of existing fields requires a major version bump.

## 9. Backward compatibility

- Legacy configs (single `upstream:` block) auto-migrate at load via `config/loader.py`.
- Entry points: `python -m sluice` and `sluice` CLI binary both work.
- Detector IDs follow `category.name` (e.g. `secrets.aws_access_key`).

## 10. Performance budget

These are gates on the v0.1 release. CI fails if benchmarks regress past them.

| Path | p50 | p95 |
|---|---|---|
| No detector hit, no taint | < 1 ms | < 3 ms |
| Detector hit, block path | < 2 ms | < 5 ms |
| Detector hit, redact path | < 4 ms | < 10 ms |
| Audit write (SQLite, batched) | < 0.5 ms | < 2 ms |

Measured on a 2023-era developer laptop, single upstream, 1 KB messages. Numbers are placeholders to be re-baselined once we have real measurements — but the discipline of *having* a budget is non-negotiable.

## 11. Architecture decisions (resolved)

| Question | Decision |
|---|---|
| CLI framework | **Typer** |
| SQLite async runtime | **aiosqlite** |
| stdio session ID | **One UUID per stdio process** |
| Policy rule order | **First-match wins** |
| Tool-poisoning scope | **Descriptions, names, and parameter schemas** |
| `${VAR}` expansion | **Config loader** (planned, not yet implemented) |
