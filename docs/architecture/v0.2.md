# Sluice v0.2 — Target Architecture

Forward-looking design contract for the next release. Written after v0.1.0 shipped.

If a code change conflicts with this doc, the doc wins — or the doc gets updated in the same PR.

**Audience:** maintainers and serious contributors.

**Status:** draft, pre-implementation. Assumes v0.1.0 as the baseline.

---

## 1. Where v0.1 landed

Shipped:
- stdio + HTTP + streamable HTTP transports
- Multi-upstream routing (`POST /u/<name>`)
- Detectors: secrets, PII, tool-poisoning
- Session-scoped taint with cross-tool leak blocking
- SQLite audit + `sluice logs`
- Typer CLI: `init`, `serve`, `stdio`, `logs`, `doctor`, `version`
- Pydantic config schema + legacy v0 migration
- `${VAR}` expansion
- Latency: 0.02 ms clean, 0.61 ms secret (pipeline only, 1 KB messages)
- CI (pytest + ruff), LICENSE, SECURITY.md, PyPI (`sluice-taint`)

Known stubs / deferrals coming into v0.2:
- Streamable HTTP SSE session lifecycle is minimal — reconnect + resume needs finishing
- No dashboard, no visual audit view
- No policy presets — every user starts from a blank config
- No integration tests against real MCP servers
- No Docker image
- Plugin entry points exist in code but are undocumented

---

## 2. v0.2 goals and non-goals

### Goals
- A security engineer at a small-to-midsize company can run Sluice in front of their team's MCP setup without writing YAML from scratch
- Policy presets for the top 5 MCP servers ship in-tree
- A read-only HTML dashboard lets a human see what Sluice blocked, redacted, and flagged, without needing to `sluice logs | jq`
- `docker run` works as a first-class deployment path
- CI runs integration tests against at least two real MCP servers
- Contributors can add a detector by reading one doc and one file

### Non-goals for v0.2 (deferred to v0.3+)
- Field-level taint provenance (v0.3)
- Prompt-injection detection on tool responses (v0.3)
- OpenTelemetry exporter (v0.3)
- Multi-tenant deployment (Sluice remains a single-team tool for v0.2)
- A configurable policy expression language beyond YAML matching
- Any hosted control plane, ever

## 4. Feature specs

### 4.1 Policy presets

**Problem:** every new user writes YAML from scratch to protect a filesystem server, then does it again for github, then again for slack. Copy-paste-paste-modify.

**Solution:** ship curated presets in-tree and let configs pull them in.

**Config surface:**

```yaml
version: 1

include:
  - "preset:github"
  - "preset:slack"
  - "./local-overrides.yaml"

upstreams:
  - name: "github"
    transport: "streamable_http"
    url: "https://api.github.com/mcp"
    headers:
      Authorization: "${GITHUB_TOKEN}"

  - name: "slack"
    transport: "streamable_http"
    url: "https://slack.com/mcp"
    headers:
      Authorization: "${SLACK_TOKEN}"
```

The `include:` list is resolved before the main config parses. `preset:<name>` looks up `sluice/policy/presets/<name>.yaml`. Local paths are relative to the config file. Later includes override earlier ones. The top-level config file overrides all includes.

**Preset format:** same schema as `config.yaml`, minus `upstreams:` (which must live in the user's file since URLs and tokens are user-specific). A preset contributes rules to `policy.rules` and defaults to `detectors`.

**CLI:**

```
sluice presets list                   # list bundled presets
sluice presets show <name>            # print the YAML
sluice presets apply <name>           # append `include: preset:<name>` to config.yaml
```

**Invariants:**
- Presets never change user-supplied `upstreams`
- Presets never override an explicit rule; user rules take precedence by position
- All presets pass a schema validation test in CI

### 4.2 Read-only HTML dashboard

**Problem:** `sluice logs` is fine for engineers with a terminal. For a security review, someone needs to *see* what's happening.

**Solution:** server-rendered HTML at `/_sluice/`, mounted on the same port as `serve`. No SPA, no build step, no JS framework.

**Routes:**

| Path | Purpose |
|---|---|
| `GET /_sluice/` | Overview: last 24h event counts by action, top firing detectors, active sessions |
| `GET /_sluice/events` | Paginated event list, filter by upstream/action/session/detector |
| `GET /_sluice/events/<id>` | Single event detail (safe preview, never raw secret) |
| `GET /_sluice/sessions` | Active sessions + taint mark count |
| `GET /_sluice/sessions/<id>` | Session detail: event timeline, taint marks (hashed) |
| `GET /_sluice/config` | Current effective config (with secrets redacted) |
| `GET /_sluice/health` | Machine-readable health for `doctor` and monitors |

**Stack:**
- Jinja2 templates (already a transitive dep via FastAPI examples; direct dep now)
- Vanilla HTML + minimal CSS in `dashboard/static/style.css`
- Zero JS frameworks. Progressive enhancement via `<details>`, `<dialog>`, `<form method="get">`
- HTMX allowed *if* we hit a real friction point; not by default

**Auth:**
- v0.2 default: bind to 127.0.0.1 only, no auth (relies on network isolation)
- Config option: `dashboard.token: "..."` — if set, all `/_sluice/*` routes require `Authorization: Bearer <token>`
- No user accounts, no cookies, no CSRF machinery. Bearer token or nothing.

**Data safety:**
- Dashboard reads from the same SQLite store as `sluice logs`
- Never renders raw matched secret text; uses `redacted_preview` from the events table
- Taint marks display as sha256 prefix + length, never the raw value

**Config:**

```yaml
dashboard:
  enabled: true
  path: "/_sluice"         # allows relocation
  token: "${SLUICE_DASHBOARD_TOKEN}"   # optional
  page_size: 50
```

Dashboard is enabled by default when running `sluice serve` on localhost. Disabled by default when binding to non-loopback addresses unless `token` is set (loader raises otherwise).

### 4.3 Docker image

**Base:** `python:3.12-slim`. Not distroless (users need `sluice doctor` and shell access for debugging).

**Image:** `ghcr.io/<repo>/sluice:0.2.0` and `ghcr.io/<repo>/sluice:latest`.

**Entrypoint:** `sluice`, no default command (user picks `serve`).

**Volumes:**
- `/etc/sluice/config.yaml` — config file
- `/var/lib/sluice/` — audit db

**Ports:** `4444`.

**Build:** GitHub Actions workflow, tag-triggered on `v*`. Multi-arch (linux/amd64, linux/arm64).

**stdio mode inside Docker:** not supported in v0.2. Docker is HTTP-only. Documented explicitly — Claude Desktop / Cursor users install via pip, not docker.

**Example:**

```bash
docker run --rm -p 4444:4444 \
  -v $PWD/config.yaml:/etc/sluice/config.yaml:ro \
  -v sluice-audit:/var/lib/sluice \
  ghcr.io/<repo>/sluice:0.2.0 serve --config /etc/sluice/config.yaml
```

### 4.4 Integration tests against real MCP servers

**Goal:** catch spec-drift bugs before users do.

**In-CI (per PR):**
- filesystem server (`@modelcontextprotocol/server-filesystem`) — pure local, no auth
- sqlite server (`mcp-server-sqlite`) — pure local, no auth
- Tests: proxy start, `tools/list`, one `tools/call` clean, one with a planted secret (blocked), one that leaks a value later (taint block)

**Nightly (out of band):**
- github MCP server with a scratch org token stored in repo secrets
- Broader coverage, longer runtime

**Test harness:** `tests/integration/` folder, pytest markers (`@pytest.mark.integration`), separate CI job. Fixtures spawn the upstream as a subprocess, wait for readiness, run the scenario, teardown.

### 4.5 Streamable HTTP SSE lifecycle

**Gap in v0.1:** SSE sessions work for the happy path but reconnect / resume-with-Last-Event-Id is stubbed.

**v0.2:** finish per the MCP 2025-03-26 spec:
- Server tracks per-session event IDs
- On reconnect, replay events after the client's `Last-Event-Id`
- Bounded per-session buffer (config: `transports.streamable_http.session_buffer: 512`)
- Session eviction after configurable idle timeout (default 5 min)

**Not doing yet:** cross-process session persistence (that's v0.3+ if we ever ship multi-worker mode).

### 4.6 CONTRIBUTING.md — detector contribution guide

Existing `CONTRIBUTING.md` covers PR basics. Extend with a walkthrough:

1. Copy `detectors/base.py`'s example detector
2. Implement `scan(text, context) -> list[Hit]`
3. Register via `@register` for in-tree, or via entry point for external package
4. Write tests in `tests/detectors/<name>.py`
5. Add a policy rule in `config.yaml.example`
6. Update `docs/detectors.md` catalogue

Also document the entry-point mechanism that already exists in v0.1 but is undocumented:

```toml
[project.entry-points."sluice.detectors"]
my_org_secrets = "my_org_sluice_extras:MyOrgSecretsDetector"
```

## 5. Config schema changes

Additive. Version stays `1` — no migration needed.

New top-level keys:
- `include: [str, ...]` — resolved before main parse
- `dashboard: {...}` — see §4.2

New nested keys:
- `transports.streamable_http.session_buffer: int` (default 512)
- `transports.streamable_http.session_idle_seconds: int` (default 300)

Unknown-key behavior unchanged: top-level unknown → error, nested unknown → warning.

## 6. Performance budget

Additive to v0.1 budgets (still enforced).

| Path | p50 | p95 |
|---|---|---|
| Dashboard overview render (last 24h, ≤10k events) | < 50 ms | < 150 ms |
| Dashboard events list (one page, 50 rows) | < 20 ms | < 80 ms |
| SSE session buffer append | < 0.1 ms | < 0.5 ms |
| SSE session reconnect + replay (100 events) | < 10 ms | < 30 ms |

Measured on the same 2023-era laptop as v0.1. CI benchmark job re-baselines each release; regressions >20% fail the build.

## 7. Backward compatibility

- All v0.1 configs continue to load unchanged.
- Dashboard is enabled-by-default on loopback binds. Users who don't want it set `dashboard.enabled: false`.
- No CLI flags removed; new `presets` command added.
- Detector protocol unchanged.
- SQLite schema — additive columns only (see §8), no destructive migration.

## 8. Audit store changes

Additive columns to `events` for dashboard filtering:

```sql
ALTER TABLE events ADD COLUMN preset_source TEXT;   -- which preset produced the matched rule, if any
ALTER TABLE events ADD COLUMN client_ip TEXT;        -- populated for HTTP transport only
```

Migration runs on first startup of v0.2. Safe on existing databases.

Retention semantics unchanged.

## 9. Open questions for v0.2

Decide before code lands. Each becomes an ADR.

1. **Dashboard framing HTML approach.** Straight Jinja + CSS, or add HTMX from day one? I lean straight Jinja for v0.2; HTMX only if we hit specific friction.
2. **Preset override precedence.** User rule wins by position or by explicit priority? Position is simpler; explicit priority is more powerful. Lean position.
3. **Docker image base.** `python:3.12-slim` vs `python:3.12-alpine` vs distroless. Lean slim — smallest legitimate footprint without breaking `sluice doctor`.
4. **Session eviction policy.** LRU vs strict idle timeout for SSE session buffers. Lean idle timeout for simplicity.
5. **Integration test framework.** pytest with subprocess fixtures, or a lightweight custom harness? Lean pytest — one less thing to learn.
6. **Nightly CI cost.** Real GitHub MCP integration test spins up billable compute. Do we accept the cost, mock at the network layer, or run these locally only? Lean: mock at the network layer for now, real integration is a v0.3 decision.

## 10. Rollout order (suggested, non-binding)

1. Streamable HTTP SSE lifecycle finish (closes v0.1 debt)
2. Integration test harness (unlocks safe refactoring for the rest)
3. Policy presets (biggest UX win per line of code)
4. CONTRIBUTING.md detector guide (unlocks external contribution)
5. Dashboard (largest surface area, land last)
6. Docker image (small, ship last as part of the v0.2 release cut)

Doing 3 and 5 in parallel is fine if you're working with your colleague. Everything else has enough coupling that serial is safer.

---

## 11. What v0.2 explicitly does not touch

Called out to prevent scope creep during implementation.

- The `Detector` protocol
- The taint algorithm (still substring, session-scoped)
- The core inspection pipeline order
- The stdio transport
- The SQLite schema beyond additive columns
- The Typer CLI's existing commands (only additions)

If any of these turn out to need changes during v0.2 build, that's a signal to pause and update this doc first.
