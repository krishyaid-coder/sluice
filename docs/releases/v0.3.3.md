# Sluice v0.3.2

Bug-fix release. Three real issues found in external review of v0.3.0, fixed with regression tests. **Recommended upgrade for all v0.3.x users**, especially anyone relying on `redact` in policy or using stdio with real MCP servers.

## Install

```bash
pip install --upgrade sluice-taint
```

PyPI: https://pypi.org/project/sluice-taint/0.3.2/

## Fixes

### 1. `redact()` no longer corrupts output when secrets and PII coexist

When a message contained both a secret and PII (say, an AWS key followed by an email address) and the matched policy action was `redact`, Sluice ran redaction in two passes with stale byte offsets. The secret got hidden correctly, but the email redaction landed at the wrong position — leaving the actual PII partially visible while covering up unrelated bytes.

Rewritten as a single unified pass over all redactable hits, sorted by position descending so each replacement only shifts bytes that have already been processed. Regression tests cover interleaved secret and PII in both orders, plus surrounding-text-untouched invariants.

### 2. Stdio upstream correlates request and response by JSON-RPC id

Previously, `StdioUpstream.forward()` did one `readline()` per request and assumed the returned line was the response. Any real MCP server that logs to stdout, sends a notification, or responds out of order would silently mispair requests and responses — potentially returning the wrong response to the agent without inspection against the correct request's context.

Refactored to a persistent reader task with an id-keyed pending-request map. Non-JSON lines, id-less notifications, and orphan responses are logged as observability events, not returned as responses. Hanging servers now time out cleanly (30 s default) instead of stalling forever. Duplicate in-flight ids and other pathological cases surface as clear errors.

### 3. Taint leaks now respect user policy

Previously, `taint_leak` detection hardcoded `action="block"` regardless of user rules. A rule like `detector: "taint_leak"` `action: "flag"` in `config.yaml` was silently ignored.

Wired through the same `resolve_action` path as every other detector. Safety rails preserve the previous safe-by-default behavior:

- No explicit taint rule (or fallback to `default_action`) → still blocks. Taint never silently inherits a permissive `default_action`.
- `action: redact` on a taint rule is rejected at config load — partial masking of the flagged value is not meaningful.
- `action: pass` is coerced to `flag` at runtime so the leak still lands in the audit trail.

## What didn't change

- All v0.3.x configs continue to work unchanged. No schema bump.
- Detector protocol, CLI surface, audit schema are all stable.
- Performance: 0.02 ms clean p50 unchanged. The redact path is now marginally faster (one loop instead of two).
- 19 new regression tests added; existing tests untouched. Full suite: 50 passing.
